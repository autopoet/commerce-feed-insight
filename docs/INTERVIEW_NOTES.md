# 电商推荐流转化分析实验台 面试讲稿

## 1. 30 秒项目介绍

这个项目叫“电商推荐流转化分析实验台”。它模拟的是广告流量进入电商推荐流之后，前端如何承接流量、渲染商品流、采集用户行为，并在本地实时形成曝光、点击、加购、购买的转化漏斗。

我做它不是为了展示一个普通虚拟列表，而是想把我之前做过的虚拟列表、图片懒加载、骨架屏能力，放到更贴近电商广告业务的场景里：广告归因、有效曝光、统一埋点和实时指标分析。

## 2. 业务链路怎么讲

可以按这条链路讲：

```text
广告入口 -> 推荐流承接 -> 商品有效曝光 -> 点击 -> 加购 -> 购买 -> 看板聚合
```

广告入口页负责选择 `channel / campaign / creative`，这些字段代表流量来源、活动和创意版本。进入推荐流之后，所有关键行为事件都会带上这几个字段，这样看板才能分析不同渠道和创意的表现。

推荐流页负责商品展示和行为触发。用户浏览商品时，只有卡片进入视口 50% 以上并停留 1000ms，才算一次有效曝光。同一个 session 内同一个商品只记一次曝光，避免用户上下滑动造成重复统计。

看板页不使用写死指标，而是从事件流里实时聚合曝光、点击、加购、购买，并计算 CTR、加购率和购买率。

## 3. 前端工程亮点

### 3.1 页面边界

项目拆成三个页面：

- 广告入口页：产生广告上下文。
- 推荐流页：承接流量并触发行为事件。
- 数据看板页：消费事件流并展示指标。

这样的拆分方便说明业务边界，也方便后续扩展路由、权限或独立模块。

### 3.2 状态设计

我用了 Zustand 保存两类状态：

- 广告上下文：`channel / campaign / creative`
- 埋点事件流：`TrackingEvent[]` 和曝光去重集合

组件不会自己拼完整事件，只传业务参数给 `track`。`track` 统一补齐 `eventId / timestamp / sessionId / userId / ad context`，这样字段口径集中，后续如果改上报策略也比较好维护。

### 3.3 性能设计

推荐流做了三层优化：

- 虚拟列表：只渲染视口附近的商品卡片，减少 DOM 数量。
- 图片懒加载：图片进入需要加载时再请求，降低首屏资源压力。
- 骨架屏：分页或等待数据时给出稳定占位，减少空白感。

如果面试官问为什么不用现成库，可以说 MVP 阶段自己实现固定高度虚拟列表，是为了能讲清楚核心原理；真实业务里会根据复杂度选择成熟虚拟列表库。

## 4. 埋点设计怎么讲

事件模型的核心是统一、可归因、可聚合。

公共字段包括：

- `eventId`
- `eventName`
- `timestamp`
- `sessionId`
- `userId`
- `page`
- `channel`
- `campaign`
- `creative`
- `productId`
- `position`
- `payload`

事件类型包括：

- `page_view`
- `product_exposure`
- `product_click`
- `add_to_cart`
- `purchase`

可以强调：看板指标全部来自事件流，避免 UI 中直接写假指标。这让前端既能负责采集，也能通过本地聚合模拟分析链路。

## 5. 有效曝光怎么讲

有效曝光不是“组件渲染了就算曝光”，而是结合了可见比例和停留时间：

- IntersectionObserver 判断卡片进入视口。
- 阈值设置为 50%。
- 满足可见后等待 1000ms。
- 如果期间离开视口，则取消计时。
- session 内同一商品只上报一次。

这个设计更贴近广告和推荐业务，因为曝光需要代表用户真正有机会看到商品。

## 6. 看板指标怎么讲

看板核心指标：

- 曝光数
- 点击数
- 加购数
- 购买数
- CTR = 点击 / 曝光
- 加购率 = 加购 / 点击
- 购买率 = 购买 / 点击

维度分析：

- 按 channel 看不同流量来源表现。
- 按 creative 看不同创意版本表现。

最近事件日志用于面试演示：点击、加购、购买之后能立刻看到事件进入日志和指标变化。

## 7. 可以主动说的取舍

这个项目为了面试和学习，没有做成企业级系统，所以做了明确取舍：

- 不接真实后端，只用 mock 数据和本地状态。
- 不接真实埋点平台，只模拟 SDK 和事件流。
- 不引入大图表库，先用轻量条形图和表格表达指标。
- 不做完整订单系统，购买只是行为闭环里的模拟事件。

这些取舍是为了把时间集中在“业务链路 + 前端工程能力 + 埋点分析思维”上。

## 8. 面试官可能追问

### 如果用户刷新页面，事件会丢吗？

当前 MVP 事件流保存在内存里，刷新会丢。可以扩展为 localStorage 持久化，或批量上报服务端。真实业务里会考虑失败重试、采样、页面卸载前 flush 和服务端去重。

### 为什么曝光按商品去重？

因为漏斗里更关心一次 session 内用户是否有效看过某个商品。如果不去重，用户来回滑动会抬高曝光，导致 CTR 被稀释。

### 购买率为什么用点击作分母？

这个项目里购买行为必须来自商品交互链路，所以 MVP 用点击作为购买转化分母。真实业务可根据分析口径扩展为购买 / 曝光、购买 / 加购，或者按订单金额计算 GMV ROI。

### 后续怎么扩展？

可以扩展：

- 接入真实接口和分页加载。
- 接入图表库做趋势分析。
- 增加 A/B 实验维度。
- 增加事件批量上报和失败重试。
- 增加商品详情页和订单链路。
- 增加性能指标采集，比如 LCP、INP、滚动帧率。

## 9. 3 分钟讲述版本

我这个项目模拟的是广告流量进入电商推荐流后的转化分析链路。入口页选择渠道、活动和创意版本，进入推荐流后，这些广告上下文字段会贯穿曝光、点击、加购、购买事件，最后在看板页实时形成转化漏斗。

推荐流部分我做了虚拟列表、图片懒加载和骨架屏。虚拟列表减少长列表 DOM 数量，懒加载降低首屏图片资源压力，骨架屏保证加载时的视觉稳定性。

埋点部分我设计了统一 `track` 方法，组件只传事件名、商品 ID、位置和 payload，`track` 统一补齐事件 ID、时间、session、用户和广告归因字段。有效曝光用 IntersectionObserver 实现，要求卡片 50% 可见并停留 1000ms，同一 session 同一商品只上报一次。

看板部分没有写死数据，而是从事件流实时聚合曝光、点击、加购、购买，并计算 CTR、加购率、购买率，还能按 channel 和 creative 看不同广告来源的表现。

所以这个项目的重点不是页面复杂度，而是用一个小型前端项目把电商推荐流、性能优化、埋点协议、广告归因和实时分析串起来。
