# 远程展示与自我熟悉脚本

## 1. 文档定位

这份文档不是现场路演稿，而是面试前自我熟悉项目、以及面试官通过 GitHub 和线上部署链接查看项目时的讲解线索。

当前实际场景是：

- 面试中大概率不能共享本地环境做完整演示。
- 面试官如果想看项目，只能通过 GitHub 仓库和部署链接访问。
- 因此项目需要做到“打开链接能看懂，听你讲能串起来”。

这份脚本的目标是帮助你在回答时不散、不乱、不只讲技术点，而是始终围绕一条业务链路：

```text
广告入口 -> 推荐流承接 -> 有效曝光 -> 点击/加购/购买 -> 实时漏斗看板
```

## 2. 面试前准备清单

面试前建议确认以下内容：

- GitHub 仓库可以打开。
- README 顶部能说明项目定位。
- 如果已经部署，README 中要放线上访问链接。
- 本地执行过 `npm run lint`、`npm run build`、`npm run test:e2e`。
- `docs/screenshots` 中有入口页、推荐流页、看板页截图。
- 自己能在 2 到 3 分钟内讲清楚项目主线。

## 3. 面试官打开项目时的推荐浏览顺序

如果面试官愿意点开 GitHub，可以引导他按这个顺序看：

1. 先看 README，了解项目不是普通商品列表，而是广告转化分析链路。
2. 再看部署链接，进入广告入口页。
3. 选择广告渠道、活动、创意，进入推荐流。
4. 在推荐流里浏览商品，点击、加购、购买。
5. 打开实时看板，看漏斗指标、渠道/创意维度和事件日志。
6. 如果想看工程设计，再看 `docs/TECH_DESIGN.md` 和 `docs/TRACKING_DESIGN.md`。

如果面试官没有时间点页面，你就直接用语言讲“这个链接里能看到什么”。

## 4. 2 分钟讲述版本

可以这样说：

> 这个项目是一个电商推荐流转化分析实验台，模拟广告流量进入推荐流后的前端链路。入口页选择广告渠道、活动和创意版本，进入推荐流后，曝光、点击、加购、购买事件都会带上这些归因字段。推荐流里我做了虚拟列表、图片懒加载和骨架屏，解决长商品流的渲染和体验问题。埋点通过统一 `track` 方法收敛，有效曝光用 50% 可见并停留 1000ms 的规则判断，同一 session 同一商品只上报一次。看板页不写死数据，而是从事件流实时计算曝光、点击、加购、购买、CTR、加购率和购买率，并按 channel 和 creative 做表现对比。

这一段是最重要的主线，先背熟它。不要一上来陷入“我用了哪些组件”。

## 5. 5 分钟讲述版本

### 第 1 分钟：项目定位

强调：

- 这不是普通虚拟列表 demo。
- 这是广告流量进入电商推荐流后的转化分析链路。
- 我把性能优化能力放进了更贴近电商广告业务的场景。

可以说：

> 我之前做过虚拟列表、懒加载、骨架屏，所以这个项目不是重新做一遍列表，而是把这些能力放到电商广告转化链路里。这样既能体现前端性能优化，也能体现我对广告归因和埋点分析的理解。

### 第 2 分钟：广告入口与归因

讲清楚三个字段：

- `channel`：流量来源，例如推荐流广告、搜索广告。
- `campaign`：活动，例如大促预热、新客转化。
- `creative`：创意版本，例如视频 A、图片 B。

重点：

- 这些字段不是只在入口页展示。
- 它们会贯穿后续所有行为事件。
- 看板可以用它们做维度对比。

### 第 3 分钟：推荐流与性能优化

讲三个点：

- 虚拟列表：只渲染视口附近商品，减少 DOM。
- 图片懒加载：降低首屏资源压力。
- 骨架屏：避免加载时空白，提升体验稳定性。

可以补一句取舍：

> MVP 阶段我自己实现固定高度虚拟列表，是为了能讲清楚原理。真实项目里如果商品卡片高度复杂，我会考虑成熟虚拟列表库。

### 第 4 分钟：埋点与有效曝光

讲统一事件模型：

- `page_view`
- `product_exposure`
- `product_click`
- `add_to_cart`
- `purchase`

讲有效曝光：

- 卡片进入视口比例大于等于 50%。
- 停留时间大于等于 1000ms。
- session 内同一商品只上报一次。

可以说：

> 我没有把组件渲染等同于曝光，因为广告和推荐业务里，曝光应该表示用户真的有机会看到这个商品。

### 第 5 分钟：实时看板

看板讲三类信息：

- 核心计数：曝光、点击、加购、购买。
- 转化率：CTR、加购率、购买率。
- 维度表现：不同 channel 和 creative 的转化差异。

重点强调：

> 看板指标全部从事件流聚合出来，不是 UI 写死数据。这样可以模拟真实业务里的“采集 -> 归因 -> 聚合 -> 分析”链路。

## 6. 面试官可能打开线上页面时的操作说明

如果 README 里有部署链接，面试官打开后可以这样体验：

1. 进入首页，选择一个广告来源。
2. 点击 `Enter Feed`。
3. 在商品推荐流里等待 1 秒，让有效曝光触发。
4. 点击商品卡片，触发点击事件。
5. 点击 `Cart`，触发加购事件。
6. 点击 `Buy`，触发购买事件。
7. 点击 `Dashboard`，查看漏斗指标变化。

如果面试官只看截图，也能看到三类页面：

- 广告入口页：说明归因上下文从哪里来。
- 推荐流页：说明性能优化和行为触发在哪里。
- 看板页：说明数据如何被聚合分析。

## 7. 被追问时的短回答

### 为什么做这个项目？

为了把已有的前端性能优化经验放进更贴近电商广告业务的场景里，不只是展示列表渲染，而是展示广告归因、埋点采集和实时转化分析链路。

### 为什么有效曝光要设置 50% 和 1000ms？

因为组件出现在 DOM 里不代表用户看到了。50% 可见加 1000ms 停留可以更接近“用户有机会看到商品”的业务口径。

### 为什么同一 session 同一商品曝光去重？

避免用户上下滑动导致同一个商品反复计入曝光，影响 CTR 等转化率判断。

### 为什么不用真实后端？

这个项目是面试和学习用 MVP，重点是前端链路设计。真实业务里事件会上报服务端，并做批量上报、失败重试、采样、服务端清洗和去重。

### 为什么不用 ECharts？

当前看板重点是说明指标口径和事件聚合，不是复杂图表展示。MVP 用轻量条形图和表格更直接；后续如果做趋势、分层、对比，可以接入 ECharts。

### 为什么不用 UI 组件库？

项目体量小，自定义样式更容易控制页面质感，也能体现基础 UI 实现能力。真实业务中会优先遵循团队设计系统。

### 真实业务还能怎么扩展？

可以接入真实商品接口、真实埋点 SDK、批量上报、A/B 实验维度、GMV/ROI 指标、推荐策略标识、性能指标采集和更完整的订单链路。

## 9. 你自己的复述检查

面试前可以用下面 5 个问题检查自己是否真的熟：

1. 我能不能一句话说清楚这个项目解决什么问题？
2. 我能不能解释广告归因字段为什么要贯穿事件？
3. 我能不能讲清楚有效曝光不是组件渲染？
4. 我能不能讲清楚看板数据是从事件流聚合出来的？
5. 我能不能说出这个项目为了 MVP 做了哪些取舍？

如果这 5 个问题都能自然回答，这个项目就已经具备面试展示价值了。
