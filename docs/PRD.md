# 电商推荐流转化分析实验台 PRD

## 1. 项目概述

### 1.1 项目名称

- 中文名：电商推荐流转化分析实验台
- 英文名：commerce-feed-insight

### 1.2 项目定位

本项目是一个面向电商交易与广告业务场景的前端实验型项目，用于模拟“广告流量进入电商推荐流后的用户转化链路”。

它不是一个单纯的虚拟列表 Demo，也不是只展示商品卡片的静态页面，而是围绕广告投放、推荐流消费、用户行为埋点、实时转化漏斗分析构建的一套小型前端业务系统。

项目希望体现前端在电商广告业务中的核心价值：

- 高效承接广告流量进入推荐流页面。
- 在复杂商品流中保障首屏速度、滚动流畅度和图片加载体验。
- 通过统一埋点模型采集曝光、点击、加购、购买等关键行为。
- 将行为数据实时汇总为转化漏斗，辅助理解不同广告渠道、活动、创意版本的表现差异。
- 用工程化拆分方式组织页面、组件、数据模型、埋点逻辑和文档。

### 1.3 项目一句话描述

一个模拟广告流量进入电商推荐流，并实时分析曝光、点击、加购、购买转化效果的 React 前端实验台。

## 2. 背景与问题

### 2.1 业务背景

在电商业务中，广告和推荐流通常不是割裂的两个模块。广告负责把用户带入交易场景，推荐流负责承接用户兴趣并持续分发商品，最终目标是提升点击、加购、下单等转化指标。

对前端来说，这类业务有几个典型特点：

- 商品流列表长，图片多，内容密集，对渲染性能和滚动性能要求高。
- 广告来源信息需要贯穿后续链路，用于判断不同渠道、活动、创意带来的转化效果。
- 用户行为事件多，曝光、点击、加购、购买都需要可追踪、可归因。
- 曝光不能简单等同于“组件渲染”，而要考虑真实进入视口并停留足够时间。
- 业务方需要实时看到漏斗变化，以便判断投放质量和商品承接效果。

### 2.2 当前要解决的问题

作为项目展示项目，如果只做虚拟列表、图片懒加载和骨架屏，容易停留在通用前端性能优化层面，业务贴合度不足。

本项目希望解决的问题是：

- 如何把已有的前端性能优化能力放进真实电商广告业务语境中。
- 如何用一个小项目讲清楚“广告归因 + 推荐流渲染 + 行为埋点 + 转化看板”的完整链路。
- 如何体现前端不仅负责页面展示，也能参与业务数据链路设计。
- 如何用工程化文档和模块拆分证明项目不是临时拼页面，而是有产品目标、技术边界和迭代路径。

### 2.3 项目价值

对项目展示而言，本项目可以支撑以下表达：

- 我理解电商广告业务中“流量进入、推荐承接、行为采集、转化分析”的基本闭环。
- 我能把虚拟列表、懒加载、骨架屏等经验放到高流量商品流场景中，而不是孤立展示技术点。
- 我能设计前端埋点模型，明确事件字段、触发时机、去重策略和归因字段。
- 我能用实时看板表达数据消费侧需求，说明前端如何帮助业务观察 CTR、加购率、购买率。

## 3. 项目目标

### 3.1 产品目标

1. 建立一个完整的广告到交易转化模拟链路。
2. 让用户可以选择广告渠道、活动和创意版本，并进入商品推荐流。
3. 在商品推荐流中模拟用户浏览、点击、加购、购买行为。
4. 将行为事件实时进入数据看板，形成可观察的转化漏斗。
5. 支持按广告渠道、活动、创意版本观察不同流量来源的表现。

### 3.2 技术目标

1. 使用 React + TypeScript + Vite 构建前端项目。
2. 商品流页面支持虚拟列表，避免大量 DOM 节点导致页面卡顿。
3. 商品图片支持懒加载和加载占位，降低首屏资源压力。
4. 页面切换和数据加载过程中提供骨架屏，优化感知体验。
5. 使用统一的 track 方法和事件模型管理埋点。
6. 使用 IntersectionObserver 或同类机制实现有效曝光判断。
7. 将业务状态、事件流和看板聚合逻辑拆分为独立模块。

### 3.3 展示目标

本项目最终服务于前端项目讲述，要求能清晰回答：

- 为什么这个项目和电商广告业务相关？
- 商品推荐流页面为什么需要虚拟列表、图片懒加载和骨架屏？
- 曝光埋点为什么不能在组件渲染时立即上报？
- 如何定义有效曝光，如何避免重复曝光？
- 广告渠道、活动、创意如何在后续事件中归因？
- 看板中的 CTR、加购率、购买率如何计算？
- 如果是真实线上业务，下一步如何接入服务端、数据平台和 A/B 实验？

## 5. 核心业务链路

### 5.1 主链路

用户从广告入口进入电商推荐流页面：

```text
广告入口选择
  -> 进入商品推荐流
  -> 商品有效曝光
  -> 点击商品
  -> 加购
  -> 模拟购买
  -> 事件进入实时看板
  -> 形成曝光、点击、加购、购买转化漏斗
```

### 5.2 归因链路

广告入口页选择的 channel、campaign、creative 会作为归因上下文贯穿整个 session。

示例：

```text
channel = douyin_feed
campaign = 618_preheat
creative = video_a
```

后续产生的 page_view、product_exposure、product_click、add_to_cart、purchase 事件都需要携带这些字段，用于在看板中按广告维度聚合。

### 5.3 数据流转

```text
用户操作
  -> UI 组件触发业务行为
  -> track 方法生成标准事件
  -> 事件写入本地事件仓库
  -> 看板订阅事件数据
  -> 计算实时指标和漏斗
  -> 页面展示统计结果与事件日志
```

MVP 阶段事件可以保存在浏览器内存或 localStorage 中，不强依赖服务端。

## 6. 页面范围

### 6.1 广告入口页

广告入口页用于模拟用户从不同广告来源进入电商推荐流。

页面能力：

- 选择广告渠道 channel。
- 选择广告活动 campaign。
- 选择创意版本 creative。
- 展示当前入口配置。
- 点击按钮进入商品推荐流。

示例选项：

- channel：抖音信息流、搜索广告、商城频道、达人短视频、外部投放。
- campaign：618 预热、夏季上新、低价秒杀、新客专享。
- creative：短视频 A、短视频 B、图文卡片 A、直播切片 A。

业务意义：

- 模拟广告投放来源。
- 为后续用户行为提供归因上下文。
- 让项目评审者直观看到广告链路入口，而不是直接进入商品列表。

### 6.2 商品推荐流页面

商品推荐流页面是项目的核心业务页面，用于承接广告流量并展示商品内容。

页面能力：

- 展示商品推荐流。
- 支持虚拟列表。
- 支持图片懒加载。
- 支持骨架屏。
- 支持分页加载或无限滚动。
- 商品卡片展示图片、标题、价格、优惠标签、销量、渠道推荐理由等信息。
- 点击商品卡片触发 product_click。
- 点击加购按钮触发 add_to_cart。
- 点击模拟购买按钮触发 purchase。
- 商品有效曝光后触发 product_exposure。

业务意义：

- 模拟真实电商推荐场景中的长列表渲染。
- 展示前端性能优化能力。
- 为曝光、点击、加购、购买提供行为发生场景。

### 6.3 实时数据看板

实时数据看板用于消费埋点事件，展示广告流量进入推荐流后的转化表现。

页面能力：

- 展示核心指标卡片。
- 展示转化漏斗。
- 展示不同 channel / creative 的表现对比。
- 展示最近事件日志。
- 支持清空当前实验数据。
- 支持回到广告入口重新开启一组实验。

业务意义：

- 将用户行为转化为业务可读指标。
- 体现前端对数据消费侧和业务分析侧的理解。
- 讲解时可以边操作边说明每个事件如何影响看板。

## 7. 功能需求

### 7.1 广告入口配置

#### 7.1.1 渠道选择

用户可以选择广告渠道。

字段：

- channelId
- channelName
- channelType

验收要求：

- 默认选中一个渠道。
- 进入推荐流后，后续事件均携带 channel 信息。
- 看板可以按 channel 聚合数据。

#### 7.1.2 活动选择

用户可以选择活动 campaign。

字段：

- campaignId
- campaignName
- campaignGoal

验收要求：

- 默认选中一个活动。
- 进入推荐流后，后续事件均携带 campaign 信息。
- 看板可以展示当前实验对应的 campaign。

#### 7.1.3 创意选择

用户可以选择创意版本 creative。

字段：

- creativeId
- creativeName
- creativeType

验收要求：

- 默认选中一个创意。
- 进入推荐流后，后续事件均携带 creative 信息。
- 看板可以按 creative 聚合点击率和购买转化率。

### 7.2 推荐流展示

#### 7.2.1 商品数据

商品数据使用本地 mock 数据生成，MVP 阶段不少于 100 条。

商品字段建议：

- productId
- title
- imageUrl
- price
- originalPrice
- discountTag
- salesCount
- shopName
- category
- recommendationReason

#### 7.2.2 商品卡片

商品卡片展示内容：

- 商品图片。
- 商品标题。
- 当前价格。
- 原价或优惠信息。
- 优惠标签。
- 销量。
- 店铺名。
- 推荐理由。
- 加购按钮。
- 模拟购买按钮。

交互要求：

- 点击卡片主体触发 product_click。
- 点击加购按钮触发 add_to_cart。
- 点击购买按钮触发 purchase。
- 按钮点击不应重复触发卡片点击事件。

#### 7.2.3 虚拟列表

商品流需要支持虚拟列表能力。

验收要求：

- 页面不一次性渲染全部商品 DOM。
- 滚动时只渲染视口附近的商品卡片。
- 长列表滚动过程保持流畅。
- 不影响曝光判断逻辑的准确性。

#### 7.2.4 图片懒加载

商品图片需要支持懒加载。

验收要求：

- 非视口附近图片不提前加载。
- 图片加载前显示占位或骨架。
- 图片加载失败时有兜底状态。

#### 7.2.5 骨架屏

页面初次加载或分页加载时需要展示骨架屏。

验收要求：

- 首次进入推荐流时展示商品卡片骨架。
- 加载更多商品时展示底部加载状态。
- 骨架屏样式与实际商品卡片结构接近，减少视觉跳动。

### 7.3 用户行为模拟

#### 7.3.1 点击商品

当用户点击商品卡片主体时，触发 product_click 事件。

要求：

- 事件携带 productId、position、广告归因字段。
- 同一商品可多次点击，多次点击均记录。
- 点击事件进入最近事件日志。

#### 7.3.2 加购

当用户点击加购按钮时，触发 add_to_cart 事件。

要求：

- 事件携带 productId、position、price。
- 加购数实时增加。
- 加购事件不触发 product_click。

#### 7.3.3 模拟购买

当用户点击购买按钮时，触发 purchase 事件。

要求：

- 事件携带 productId、position、price、orderAmount。
- 购买数实时增加。
- 购买转化率实时更新。
- MVP 阶段不需要真实支付流程。

## 8. 埋点需求

### 8.1 统一事件模型

项目需要设计统一 track 方法，所有用户行为通过 track 上报。

事件字段建议：

```ts
type TrackingEvent = {
  eventId: string;
  eventName:
    | 'page_view'
    | 'product_exposure'
    | 'product_click'
    | 'add_to_cart'
    | 'purchase';
  timestamp: number;
  sessionId: string;
  userId: string;
  page: 'ad_entry' | 'feed' | 'dashboard';
  channel?: string;
  campaign?: string;
  creative?: string;
  productId?: string;
  position?: number;
  payload?: Record<string, unknown>;
};
```

### 8.2 事件类型

#### 8.2.1 page_view

触发时机：

- 进入广告入口页。
- 进入推荐流页。
- 进入实时看板页。

用途：

- 记录页面访问。
- 确认广告归因上下文是否正确传递。

#### 8.2.2 product_exposure

触发时机：

- 商品卡片进入视口比例大于等于 50%。
- 且连续停留时间大于等于 1000ms。

去重要求：

- 同一个 session 内，同一个 productId 只上报一次有效曝光。

用途：

- 计算曝光数。
- 作为 CTR 分母。
- 判断推荐流承接广告流量后的商品可见情况。

#### 8.2.3 product_click

触发时机：

- 用户点击商品卡片主体区域。

用途：

- 计算点击数。
- 作为 CTR 分子。
- 分析不同广告渠道和创意带来的兴趣质量。

#### 8.2.4 add_to_cart

触发时机：

- 用户点击加购按钮。

用途：

- 计算加购数。
- 计算加购转化率。
- 判断用户是否从兴趣进入交易意图。

#### 8.2.5 purchase

触发时机：

- 用户点击模拟购买按钮并完成模拟购买。

用途：

- 计算购买数。
- 计算购买转化率。
- 判断广告流量最终交易效果。

### 8.3 有效曝光规则

有效曝光定义：

```text
商品卡片进入视口比例 >= 50%，并且连续停留时间 >= 1000ms。
```

规则说明：

- 组件被渲染不等于曝光。
- 快速滑过商品不应计为有效曝光。
- 已曝光商品在当前 session 内不重复上报。
- 虚拟列表导致商品组件卸载后重新挂载时，也不能重复上报同一商品曝光。

实现建议：

- 使用 IntersectionObserver 监听商品卡片可见比例。
- 当 intersectionRatio >= 0.5 时启动计时器。
- 当比例低于 0.5 或组件卸载时清除计时器。
- 计时达到 1000ms 后调用 track。
- 使用 Set 记录当前 session 已曝光 productId。

### 8.4 归因字段传递

从广告入口页进入推荐流时，需要保存广告上下文。

广告上下文字段：

- channel
- campaign
- creative

传递方式：

- MVP 阶段可使用 URL query、React Router state、Zustand store 或 localStorage。
- 后续事件均从统一上下文读取归因字段。

验收要求：

- 从不同广告入口进入推荐流时，看板能区分来源表现。
- 所有核心事件都包含当前广告归因字段。

## 9. 数据看板需求

### 9.1 核心指标

看板需要展示以下指标：

- 曝光数 exposures
- 点击数 clicks
- 加购数 addToCarts
- 购买数 purchases
- CTR
- 加购转化率
- 购买转化率

### 9.2 指标计算口径

CTR：

```text
CTR = 点击数 / 曝光数
```

加购转化率：

```text
加购转化率 = 加购数 / 点击数
```

购买转化率：

```text
购买转化率 = 购买数 / 点击数
```

当分母为 0 时，展示为 0%，避免 NaN 或 Infinity。

### 9.3 漏斗展示

漏斗层级：

```text
曝光 -> 点击 -> 加购 -> 购买
```

展示要求：

- 每一层展示数量。
- 每一层展示相对上一层的转化率。
- 通过简单柱状图、漏斗图或自定义 UI 表达层级递减关系。

### 9.4 维度对比

看板需要展示不同广告维度的表现。

至少支持：

- 按 channel 对比。
- 按 creative 对比。

展示字段：

- 曝光数
- 点击数
- 加购数
- 购买数
- CTR
- 购买转化率

MVP 阶段可以使用表格或轻量图表，不强制引入复杂可视化库。

### 9.5 最近事件日志

最近事件日志展示最新产生的行为事件。

展示字段：

- 时间
- 事件类型
- 页面
- 商品 ID
- 广告渠道
- 活动
- 创意

要求：

- 最新事件展示在最上方。
- 最多展示最近 20 条。
- 事件名称使用中文可读文案。

## 10. 性能与体验需求

### 10.1 首屏体验

要求：

- 广告入口页快速可交互。
- 推荐流页面初始加载时展示骨架屏。
- 商品图片加载不阻塞基础布局展示。

### 10.2 滚动性能

要求：

- 商品列表使用虚拟列表减少 DOM 数量。
- 滚动过程中避免频繁 setState 导致大范围重渲染。
- 曝光监听逻辑需要控制副作用范围，避免对每一帧滚动都做重计算。

### 10.3 图片体验

要求：

- 商品图片懒加载。
- 图片区域有固定比例，避免加载后布局跳动。
- 加载失败时展示兜底背景或占位文案。

### 10.4 交互反馈

要求：

- 加购后按钮或商品卡片有轻量反馈。
- 购买后给出成功反馈。
- 看板数据变化应及时更新。
- 清空数据时需要明确反馈。

### 10.5 移动端适配

项目应优先保证桌面端项目展示体验，同时兼顾移动端基础可用。

要求：

- 商品卡片在窄屏下保持可读。
- 看板指标卡片可换行。
- 入口配置表单在移动端不横向溢出。

## 15. 验收标准

### 15.1 业务链路验收

- 用户可以从广告入口选择 channel、campaign、creative。
- 用户可以进入商品推荐流页面。
- 推荐流可以展示商品列表。
- 商品有效曝光后，看板曝光数增加。
- 点击商品后，看板点击数增加。
- 加购商品后，看板加购数增加。
- 模拟购买后，看板购买数增加。
- CTR、加购转化率、购买转化率计算正确。
- 最近事件日志展示最新行为。

### 15.2 埋点验收

- 所有核心行为都通过统一 track 方法生成事件。
- 每个事件都有 eventId、eventName、timestamp、sessionId、userId、page。
- 广告归因字段能贯穿推荐流和看板。
- 同一 session 内同一商品只上报一次 product_exposure。
- product_click、add_to_cart、purchase 不做同商品去重。

### 15.3 性能体验验收

- 推荐流不一次性渲染全部商品 DOM。
- 图片懒加载生效。
- 初始加载有骨架屏。
- 滚动商品流时没有明显卡顿。
- 商品图片加载不会造成明显布局跳动。

- docs/PRD.md 存在且内容完整。
- 后续补充 docs/TECH_DESIGN.md 和 docs/TRACKING_DESIGN.md。

## 16. 项目讲述重点

### 16.1 开场介绍

可以这样介绍项目：

```text
这个项目叫电商推荐流转化分析实验台，核心不是做一个普通商品列表，而是模拟广告流量进入电商推荐流后的完整转化链路。用户从广告入口选择渠道、活动和创意版本，进入商品推荐流后产生曝光、点击、加购和购买行为，前端通过统一埋点模型采集事件，并在实时看板里形成转化漏斗和渠道表现对比。
```

### 16.2 业务理解重点

需要强调：

- 广告和推荐流在电商场景中是连续链路。
- 前端承接的不只是页面展示，还有性能体验和行为数据采集。
- 曝光、点击、加购、购买对应从流量到交易的逐层转化。
- channel、campaign、creative 是广告效果分析的关键归因字段。

### 16.3 技术实现重点

需要强调：

- 长商品流用虚拟列表降低 DOM 压力。
- 图片懒加载降低首屏资源压力。
- 骨架屏改善加载感知。
- IntersectionObserver 结合停留时间实现有效曝光。
- Set 记录已曝光商品，解决虚拟列表重复挂载带来的重复曝光问题。
- track 方法统一事件模型，避免埋点散落。
- 看板由事件流聚合得出，而不是手写假数据。

### 16.4 可扩展性重点

可以说明：

- 当前数据在前端本地模拟，真实业务中可以替换为埋点 SDK 和后端实时计算。
- 当前商品是 mock 数据，真实业务中可以替换为推荐接口。
- 当前归因是 session 级，后续可扩展到多触点归因和 A/B 实验。
- 当前看板是轻量前端聚合，后续可接入 BI 或实时数仓。

### 16.5 技术交流中可主动抛出的思考

可以主动补充：

- 如果曝光量很大，前端埋点需要批量上报、节流、失败重试和页面卸载前 flush。
- 如果商品卡片高度不固定，虚拟列表需要处理动态高度缓存。
- 如果要保证数据准确性，前端曝光只能作为采集侧，最终口径仍需要服务端清洗和去重。
- 如果不同广告创意带来的 CTR 高但购买率低，可能说明创意吸引点击但商品承接或人群质量存在问题。
