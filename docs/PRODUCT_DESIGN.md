# 电商推荐流转化分析实验台 产品设计文档

## 1. 文档定位

这份文档是 `commerce-feed-insight` 的产品设计约束文档，用来指导后续页面、交互、状态和代码实现。

项目不是企业级系统，也不追求大而全。它的目标是在短周期快速做出一个能跑、能讲、能体现业务理解和工程化意识的小型前端项目。

因此本项目的产品设计原则是：

- 够贴近业务，但不模拟完整电商平台。
- 够工程化，但不堆企业级复杂流程。
- 够可展示，但不做过度包装。
- 够可扩展，但 MVP 优先跑通主链路。

后续每次新增功能或修改交互，都需要同步更新本文档。每次写代码前，都要先读一遍本文档，确认当前实现没有偏离产品设计。

### 1.1 当前产品状态

当前 MVP 已完成并部署到 Vercel：

- 在线预览：https://commerce-feed-insight.vercel.app/
- 页面语言：中文 UI，保留必要英文技术缩写，如 `CTR`、`channel`、`campaign`、`creative`。
- 页面范围：广告入口页、商品推荐流页、实时数据看板页。
- 交互链路：选择广告入口 -> 进入推荐流 -> 触发曝光/点击/加购/购买 -> 查看实时看板。
- 验证方式：Playwright 冒烟测试覆盖主链路，并更新 `docs/screenshots` 中的页面截图。

## 2. 项目设计目标

### 2.1 学习目标

通过这个项目理解电商广告业务中的基本链路：

```text
广告入口 -> 推荐流承接 -> 用户行为 -> 埋点事件 -> 转化漏斗
```

重点理解：

- 为什么广告来源需要被归因。
- 为什么曝光需要有效曝光规则。
- 为什么推荐流需要性能优化。
- 为什么看板要由事件流聚合，而不是写死数据。

### 2.2 展示目标

讲解时需要能在 3 到 5 分钟内讲清楚：

- 我做的不是普通商品列表，而是广告流量进入推荐流后的转化分析链路。
- 我用虚拟列表、懒加载、骨架屏解决商品流性能和体验问题。
- 我设计了统一 track 方法，让曝光、点击、加购、购买都能被追踪。
- 我把 channel、campaign、creative 作为广告归因字段贯穿后续事件。
- 我用实时看板展示 CTR、加购率、购买率，让前端具备业务分析表达。

### 2.3 工程目标

项目需要有清晰的最小工程结构：

- 页面拆分清晰。
- 组件职责明确。
- 类型定义统一。
- mock 数据集中维护。
- 埋点逻辑不要散落在每个组件里。
- 指标计算逻辑独立于 UI。

不追求：

- 完整后端。
- 复杂权限。
- 企业级配置平台。
- 大量图表。
- 完整订单系统。

## 3. 产品体验基调

本项目是一个“电商推荐流转化分析实验台”，界面应更像运营分析工具，而不是营销官网。

设计基调：

- 信息密度适中，方便项目演示时快速扫读。
- 页面层级清楚，用户能知道自己在哪个链路阶段。
- 少用大段解释文案，更多通过数据、状态和事件日志表达业务。
- 视觉上偏克制、专业、轻量，不做夸张动效。
- 保留必要的操作反馈，让演示过程有变化。

界面关键词：

- 清晰
- 数据感
- 电商感
- 可演示
- 轻工程化

MVP 阶段包含三个核心页面：

/dashboard      实时数据看板页

## 5. 全局信息架构

全局需要维护三类信息：

### 5.1 广告上下文 AdContext

由广告入口页选择产生。

字段：

```ts
type AdContext = {
  channel: string;
  campaign: string;
  creative: string;
};
```

设计规则：

- 进入推荐流前必须有默认广告上下文。
- 推荐流和看板都读取同一份广告上下文。
- 所有核心事件都要带上广告上下文。
- MVP 阶段可以使用 Zustand、React Context 或 localStorage，优先选择实现简单、讲述清楚的方式。

### 5.2 事件流 TrackingEvent[]

由统一 track 方法写入。

事件类型：

- page_view
- product_exposure
- product_click
- add_to_cart
- purchase

设计规则：

- 看板数据必须从事件流计算得出。
- 最近事件日志直接消费事件流。
- 不在看板里手写假指标。

### 5.3 商品数据 Product[]

MVP 使用本地 mock 数据。

设计规则：

- 商品数量不少于 100 条，用于体现长列表场景。
- 商品字段要服务推荐流展示，不模拟复杂库存。
- 商品数据集中放在 mock 文件中，不散落在组件里。

## 6. 页面设计

### 6.1 广告入口页

#### 页面目标

让用户选择一次广告流量来源，并进入推荐流。

它的意义不是做复杂投放配置，而是告诉项目评审者：后续所有行为都不是孤立发生的，而是带着广告归因上下文发生的。

#### 页面结构

```text
顶部项目标题
  - 电商推荐流转化分析实验台
  - 简短副标题：模拟广告流量进入推荐流后的转化链路

广告入口配置区
  - 渠道 channel
  - 活动 campaign
  - 创意 creative

当前归因预览
  - 展示 channel / campaign / creative

主操作
  - 进入推荐流

辅助入口
  - 查看实时看板
```

#### 交互规则

- 页面加载时自动选中默认 channel、campaign、creative。
- 用户切换选项后，归因预览立即更新。
- 点击“进入推荐流”后：
  - 保存广告上下文。
  - 记录 page_view 或进入 feed 后记录 page_view。
  - 跳转到推荐流页。
- 点击“查看实时看板”可直接进入看板，用于讲解时快速展示统计结果。

#### 文案方向

入口页文案要短，不做长篇业务说明。

推荐主文案：

```text
选择一次广告入口，观察它在推荐流中的后续转化。
```

#### 项目讲述点

这里可以讲：

- 广告入口选择不是装饰，而是为了后续归因。
- channel、campaign、creative 会贯穿曝光、点击、加购、购买事件。

### 6.2 商品推荐流页

#### 页面目标

承接广告流量，展示商品推荐流，并产生用户行为事件。

#### 页面结构

```text
顶部状态栏
  - 当前广告来源
  - 去看板按钮
  - 返回入口按钮

推荐流主体
  - 商品虚拟列表
  - 商品卡片
  - 骨架屏 / 加载更多状态

轻量操作反馈
  - 加购成功反馈
  - 购买成功反馈
```

#### 商品卡片结构

```text
商品图片
商品标题
价格 / 原价
优惠标签
销量
推荐理由
加购按钮
购买按钮
```

#### 交互规则

- 商品卡片进入有效曝光条件后，触发 product_exposure。
- 点击商品卡片主体，触发 product_click。
- 点击加购按钮，触发 add_to_cart。
- 点击购买按钮，触发 purchase。
- 加购和购买按钮需要阻止事件冒泡，避免同时触发 product_click。
- 操作后给出轻量反馈，例如按钮文案短暂变化或页面 toast。

#### 有效曝光规则

有效曝光定义：

```text
商品卡片进入视口比例 >= 50%，并连续停留 >= 1000ms。
```

设计规则：

- 组件渲染不等于曝光。
- 快速划过不算曝光。
- 同一 session 内同一 productId 只曝光一次。
- 虚拟列表导致组件卸载后再挂载，也不能重复曝光。

#### 推荐流性能规则

- MVP 必须体现虚拟列表思想。
- 图片使用懒加载。
- 首次加载展示骨架屏。
- 商品图片区域固定比例，避免布局跳动。
- 不为了视觉效果牺牲滚动体验。

#### 项目讲述点

这里可以讲：

- 推荐流是广告流量承接页，性能体验会影响后续转化。
- 有效曝光必须基于真实可见和停留时间。
- 虚拟列表和曝光统计之间有冲突点，需要用 session 级 Set 去重。

### 6.3 实时数据看板页

#### 页面目标

把事件流转化成可读的业务指标，展示广告流量在推荐流中的转化效果。

#### 页面结构

```text
顶部状态栏
  - 当前实验上下文
  - 返回推荐流
  - 重新选择广告入口
  - 清空数据

核心指标区
  - 曝光数
  - 点击数
  - 加购数
  - 购买数
  - CTR
  - 加购转化率
  - 购买转化率

转化漏斗区
  - 曝光 -> 点击 -> 加购 -> 购买

维度对比区
  - 按 channel 对比
  - 按 creative 对比

最近事件日志
  - 最新 20 条事件
```

#### 指标计算规则

```text
CTR = 点击数 / 曝光数
加购转化率 = 加购数 / 点击数
购买转化率 = 购买数 / 点击数
```

规则：

- 分母为 0 时展示 0%。
- 百分比保留 1 位小数即可。
- 指标来自事件流聚合，不写死。

#### 交互规则

- 进入看板时记录 page_view。
- 清空数据时清空事件流和已曝光集合。
- 清空后指标恢复为 0。
- 事件日志最新事件展示在最上方。

#### 项目讲述点

这里可以讲：

- 看板证明前端采集的数据可以被业务消费。
- CTR 反映广告流量带来的兴趣程度。
- 加购率和购买率反映交易意图和最终转化。
- 如果 CTR 高但购买率低，可以分析创意、商品承接、人群匹配等问题。

## 7. 核心用户流程

### 7.1 标准演示流程

```text
打开广告入口页
  -> 选择“抖音信息流 / 618 预热 / 短视频 A”
  -> 点击进入推荐流
  -> 等待部分商品有效曝光
  -> 点击一个商品
  -> 加购一个商品
  -> 模拟购买一个商品
  -> 进入实时看板
  -> 查看漏斗、指标和事件日志
```

### 7.2 二次实验流程

```text
返回广告入口页
  -> 切换 creative 或 channel
  -> 再次进入推荐流
  -> 产生新事件
  -> 在看板中对比不同来源表现
```

MVP 阶段可以先保证单次实验顺畅，二次实验作为优先增强项。

## 8. 埋点产品规则

### 8.1 track 方法定位

track 是项目中的统一事件入口。

产品规则：

- UI 组件不直接拼复杂事件对象。
- UI 组件只传递必要业务参数。
- track 自动补齐 eventId、timestamp、sessionId、userId、广告上下文等公共字段。

### 8.2 事件展示文案

事件日志中使用中文文案：

```text
page_view -> 页面访问
product_exposure -> 商品曝光
product_click -> 商品点击
add_to_cart -> 加购
purchase -> 购买
```

### 8.3 事件必要字段

所有事件必须包含：

- eventId
- eventName
- timestamp
- sessionId
- userId
- page

商品相关事件额外包含：

- productId
- position

归因相关字段：

- channel
- campaign
- creative

## 9. MVP 实现边界

### 9.1 必须实现

- 三个页面。
- 广告上下文选择与传递。
- 商品推荐流。
- 虚拟列表。
- 图片懒加载。
- 骨架屏。
- 五类埋点事件。
- 有效曝光与曝光去重。
- 实时指标看板。
- 最近事件日志。

### 9.2 可以简化

- 图表可以先用自定义条形图，不急着引入 ECharts。
- 商品详情页不做。
- 购买只做按钮模拟。
- 数据只保存在前端本地。
- 用户 ID 和 session ID 可以本地生成。
- 推荐算法用 mock 推荐理由代替。

### 9.3 暂不实现

- 真实登录。
- 真实支付。
- 真实后端接口。
- 复杂 A/B 实验。
- 企业级埋点 SDK。
- 完整 BI 分析系统。
